# FakturaBot PDF Layout Specification

## 1. Призначення

Цей документ фіксує **layout/design вимоги** для PDF-фактури у FakturaBot як окремий артефакт наступної фази реалізації.

Мета PDF у цьому продукті — не просто вивести дані, а сформувати охайний, сучасний і діловий документ, який:
- створює wow-ефект MVP;
- придатний для реального надсилання клієнту;
- легко читається на екрані ПК, на телефоні та після друку.

Це **не специфікація коду генератора**, а саме візуально-структурний орієнтир для майбутньої реалізації.

---

## 2. Дизайн-принципи

PDF має відповідати таким принципам:

- **Clean business invoice**: професійний рахунок без зайвого декору.
- **Subtle two-tone layout**: стримана двотональна подача блоків.
- **High readability**: реквізити та суми читаються з першого погляду.
- **Visual hierarchy first**: пріоритет логіки сприйняття (header → сторони → таблиця → total).
- **No flashy styling**: без яскравих контрастів та агресивних акцентів.
- **No rainbow / no overdesign**: без перевантаження кольорами та декором.
- **Modern but restrained**: сучасно, але стримано й ділово.

---

## 3. Кольорові принципи

Кольорова система має бути практичною та стабільною:

- основний фон сторінки — білий;
- другий фон — дуже м’який нейтральний або холодний світлий тон (для секцій/рядків);
- один темніший спокійний акцент — для заголовків і/або блоку total;
- заборонено яскраві “кричущі” кольори;
- у межах одного документа використовувати не більше **2–3 тонів**.

Обов’язкова вимога: PDF має використовувати **два приємні фонові тони**, але **без перевантаження дизайну**.

---

## 4. Блоки layout (обов’язковий порядок)

Нижче — обов’язковий порядок і роль блоків.

### 4.1 Header

- Назва документа: `Faktúra`.
- Номер faktúry (виразно, поруч із заголовком або під ним).
- Верхній блок повинен бути візуально сильним.
- Допускається тонований фон для header section.

### 4.2 Supplier / Customer block

- Дві чіткі колонки:
  - `Dodávateľ`
  - `Odberateľ`
- Блокова структура з достатніми відступами.
- Кожен реквізит — окремим читабельним рядком.
- Не змішувати дані контрагентів у суцільний текст.

### 4.3 Meta / Dates block

Окремий компактний блок метаданих:

- `Dátum vystavenia`
- `Dátum dodania`
- `Dátum splatnosti`
- `Variabilný symbol`
- `Spôsob úhrady` (якщо поле присутнє у моделі)

Блок має виглядати як структурований список пар “label/value”.

### 4.4 Payment block

Окремий від supplier/customer блок реквізитів оплати:

- IBAN
- SWIFT/BIC
- інші платіжні реквізити за потреби

Принцип: платіжні поля не мають губитись серед адресних реквізитів.

### 4.5 Items table

Таблиця позицій — центральна частина документа:

- шапка таблиці з м’яким контрастним фоном;
- обов’язкові колонки:
  - `položka`
  - `množstvo`
  - `m.j.`
  - `cena za m.j.`
  - `spolu`
- навіть для однієї позиції table layout має виглядати повноцінно;
- структура таблиці повинна бути готова до розширення на кілька рядків у майбутньому.

### 4.6 Total block

Окремий сильний акцентний блок:

- `Na úhradu`
- підсумкова сума

Це один із ключових візуальних акцентів PDF.

### 4.7 QR block

- QR Pay by Square.
- Розміщення: поруч із total block або безпосередньо під ним.
- Позиціювання має бути логічним і не руйнувати загальну композицію.

### 4.8 Footer / note

- Коротка примітка або footer-текст (опційно).
- Не перевантажувати нижню частину сторінки зайвими деталями.

---

## 5. Семантика дат

Щоб layout не суперечив продуктовій логіці, фіксуються такі правила:

- `Dátum vystavenia` = дата створення faktúry (ставиться автоматично ботом).
- Якщо користувач назвав дату у voice/text input, ця дата трактується як `Dátum dodania`.
- Якщо дата не названа, тоді `Dátum dodania = Dátum vystavenia`.
- `Dátum splatnosti = Dátum vystavenia + due_days`.

---

## 6. Правило preview / approval

PDF у цьому сценарії генерується для перевірки користувачем перед фіналізацією.

Після генерації фактура проходить один із кроків:
- `schváliť`
- `upraviť`

Email send не входить у цей layout spec, але сам layout має бути придатний до майбутнього email workflow.

---

## 7. Типографіка та відступи

Типографіка й ритм документа мають залишатися простими:

- максимум 1–2 шрифтові ролі;
- чітка ієрархія розміру:
  - title,
  - section headings,
  - body,
  - total amount;
- достатньо “повітря” між секціями;
- уникати зліпленого, щільного документа без відступів.

---

## 8. Чого не робити

Що заборонено в цьому layout-підході:

- не копіювати наявний зразок 1-в-1;
- не робити plain-text dump без ієрархії;
- не використовувати яскраву “веселку” кольорів;
- не перевантажувати декоративними елементами;
- не змішувати занадто багато стилів в одному PDF;
- не робити layout, що добре виглядає лише на великому моніторі.

---

## 9. Мінімальна імплементаційна примітка

Практична примітка для наступної фази:

- цей layout spec підготовлено для подальшої реалізації в `reportlab`;
- перша реалізація повинна підтримати одну позицію в таблиці;
- архітектурно layout не повинен блокувати майбутнє розширення на кілька позицій.
